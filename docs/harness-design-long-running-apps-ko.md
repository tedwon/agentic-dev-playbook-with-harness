# 장기 실행 애플리케이션 개발을 위한 하네스 설계

> 원문: [Harness Design for Long-Running Application Development](https://www.anthropic.com/engineering/harness-design-long-running-apps)
> 저자: Prithvi Rajasekaran (Anthropic Labs) | 발행일: 2026년 3월 24일

## 한눈에 보기

이 글은 Claude가 **복잡하고 오래 걸리는 작업**(프론트엔드 디자인, 풀스택 앱 개발 등)을 수행할 때, 어떻게 하면 더 좋은 결과를 낼 수 있는지를 다룹니다. 핵심 아이디어는 **"만드는 에이전트"와 "평가하는 에이전트"를 분리**하는 것입니다.

---

## 1. 문제: AI 에이전트가 혼자서 잘 못하는 것들

AI 에이전트에게 복잡한 앱을 통째로 만들어달라고 하면 두 가지 문제가 반복적으로 발생합니다.

### 1-1. 컨텍스트 저하 (Context Degradation)

대화가 길어질수록 모델이 앞부분의 내용을 잊거나 일관성을 잃습니다. 심한 경우 **"컨텍스트 불안"** 이 생겨서 아직 할 일이 남았는데 "다 끝났다"고 성급하게 마무리하려 합니다.

> **비유:** 책을 읽다가 500페이지쯤 되면 1장 내용이 가물가물한 것과 같습니다.

### 1-2. 자기 평가 편향 (Self-Evaluation Bias)

에이전트가 자기가 만든 결과물을 스스로 평가하면, 대부분 **"잘했다"** 라고 후하게 점수를 줍니다. 특히 디자인처럼 주관적인 작업에서 이 현상이 심합니다.

> **비유:** 자기가 쓴 글을 자기가 교정하면 오타를 못 찾는 것과 같습니다.

## 2. 해결책: GAN에서 영감을 얻은 생성자-평가자 구조

이미지 생성 AI인 **GAN(Generative Adversarial Network)** 에서 아이디어를 가져왔습니다.

```
┌─────────────┐         ┌─────────────┐
│   생성자      │  작업물 → │   평가자      │
│ (Generator)  │ ←피드백── │ (Evaluator)  │
└─────────────┘         └─────────────┘
       ↑                        │
       └── 개선 반복 (5~15회) ───┘
```

- **생성자(Generator):** 코드를 작성하거나 디자인을 만드는 에이전트
- **평가자(Evaluator):** 결과물을 실제로 테스트하고 구체적인 피드백을 주는 별도의 에이전트

### 왜 분리해야 하나?

평가자도 LLM이라서 처음에는 너그럽게 평가하는 성향이 있습니다. 하지만 **독립된 평가자를 엄격하게 튜닝하는 것**이, 생성자에게 "자기 비판을 잘 하라"고 시키는 것보다 **훨씬 쉽습니다**.

## 3. 실제 적용 사례

### 3-1. 프론트엔드 디자인

평가자가 실제 UI를 **Playwright MCP**로 조작하면서 스크린샷을 찍고, 4가지 기준으로 점수를 매깁니다.

| 기준 | 설명 |
|------|------|
| **디자인 품질** | 색상, 타이포그래피, 레이아웃의 조화 |
| **독창성** | 뻔한 템플릿이 아닌 고유한 디자인 결정 |
| **완성도** | 자간, 여백, 대비 등 세밀한 기술적 실행 |
| **기능성** | 사용성과 작업 완수 가능 여부 |

이 반복 과정에서 한 박물관 웹사이트는 평범한 다크 테마에서 시작해, **CSS로 구현된 3D 갤러리 공간**이라는 혁신적인 디자인으로 진화했습니다.

### 3-2. 풀스택 코딩 - 3단계 에이전트 시스템

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  기획자    │ ──→ │  생성자    │ ──→ │  평가자    │
│ (Planner) │     │(Generator)│     │(Evaluator)│
└──────────┘     └──────────┘     └──────────┘
 사양서 작성      코드 구현         실제 테스트
```

| 에이전트 | 역할 |
|---------|------|
| **기획자 (Planner)** | 간단한 프롬프트를 야심찬 제품 사양서로 변환. AI 기능을 자연스럽게 설계에 통합 |
| **생성자 (Generator)** | React, Vite, FastAPI 등으로 실제 코드 구현. Git으로 버전 관리. 자체 1차 검증 |
| **평가자 (Evaluator)** | Playwright MCP로 실제 사용자처럼 앱을 테스트. 스프린트 계약서 대비 검증 |

## 4. 비교 실험: 레트로 비디오 게임 메이커

같은 "레트로 비디오 게임 메이커" 앱을 혼자 만든 경우와 하네스 시스템으로 만든 경우를 비교했습니다.

| 항목 | 에이전트 단독 | 하네스 시스템 |
|------|:----------:|:----------:|
| **소요 시간** | 20분 | 6시간 |
| **비용** | $9 | $200 |
| **기능 수** | 겉보기만 완성 | 16개 기능, 10개 스프린트 |
| **실제 작동** | 핵심 기능 고장 | 실제 플레이 가능 |
| **AI 통합** | 없음 | 있음 |

> **핵심 교훈:** 단독 에이전트는 **겉보기에 그럴듯하지만 실제로 작동하지 않는** 결과물을 만듭니다. 하네스 시스템은 시간과 비용이 더 들지만, **진짜 동작하는 완성도 높은** 앱을 만듭니다.

## 5. 하네스의 진화: 불필요한 복잡성 제거

모델 성능이 올라가면서(Opus 4.6), 저자는 체계적으로 구성 요소를 제거하면서 **어떤 것이 진짜 필요한지** 확인했습니다.

### 발견한 사실들

| 구성 요소 | 결론 |
|----------|------|
| 스프린트 기반 분할 | 최신 모델에서는 **불필요** |
| 평가자 (QA) | 작업 난이도에 따라 **선택적** |
| 컨텍스트 리셋 | 컨텍스트 요약(compaction)보다 **효과적** |

### 핵심 원칙

> **"하네스의 모든 구성 요소는 '모델이 혼자서 못하는 것'에 대한 가정을 담고 있다. 그 가정은 주기적으로 검증할 가치가 있다."**

즉, 모델이 발전하면 하네스도 함께 간소화해야 합니다. 과거에 필요했던 보조 장치가 지금은 오히려 방해가 될 수 있습니다.

## 6. 평가자가 잡아낸 실제 버그 사례

평가자가 발견한 버그들은 매우 구체적이고 바로 수정 가능한 수준이었습니다.

| 발견된 문제 | 설명 |
|-----------|------|
| 사각형 채우기 도구 오류 | 영역을 채우는 대신 끝점에만 타일을 배치 |
| 삭제 핸들러 논리 모순 | 조건문이 서로 충돌하여 삭제가 작동 안 함 |
| FastAPI 라우트 순서 문제 | `"reorder"` 경로가 정수 ID로 파싱됨 |

> 이런 버그들은 생성자 에이전트가 자체 검증으로는 발견하지 못한 것들입니다.

## 7. 실전 가이드라인

### 컨텍스트 관리: 요약보다 리셋

```
❌ 컨텍스트 요약 (Compaction)
   → 대화를 요약해서 이어가기
   → 모델이 "곧 끝나야 한다"는 불안감을 느낌

✅ 컨텍스트 리셋 (Context Reset)
   → 대화를 완전히 새로 시작하되, 구조화된 핸드오프 문서를 제공
   → 모델이 신선한 상태에서 집중력을 유지
```

### 주관적 기준을 구체적 기준으로 변환

```
❌ "디자인을 예쁘게 만들어줘"
   → 모호해서 평가 불가

✅ "색상 조화 / 타이포그래피 위계 / 여백 일관성 / 대비 비율을 기준으로 평가"
   → 측정 가능한 기준으로 생성자와 평가자 모두 개선
```

### 하네스 복잡도를 모델 능력에 맞추기

```
모델 능력 낮음 → 하네스 복잡도 높음 (다단계 에이전트, 스프린트 분할, QA 필수)
모델 능력 높음 → 하네스 복잡도 낮음 (단순 구조, QA 선택적)
```
