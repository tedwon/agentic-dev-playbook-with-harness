# 효과적인 AI Agent 구축하기

> 원문: [Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents) — Anthropic (2024.12.19)

## 핵심 메시지

> "가장 성공적인 구현은 복잡한 Framework가 아니라, **단순하고 조합 가능한 패턴**을 사용한다."

이 문서는 Anthropic이 고객사들과 Agent를 구축한 경험을 바탕으로 정리한 실전 가이드예요.
핵심은 **필요 이상으로 복잡하게 만들지 말라**는 것이에요.

---

## 1. Agentic System이란?

AI를 활용한 시스템은 크게 두 가지로 나눌 수 있어요:

| 유형 | 설명 | 비유 |
|------|------|------|
| **Workflow** | 미리 정해진 코드 경로를 따라 LLM과 도구가 동작 | 조립 라인 — 순서가 정해져 있음 |
| **Agent** | LLM이 스스로 판단해서 프로세스와 도구 사용을 제어 | 숙련된 작업자 — 상황에 따라 스스로 판단 |

### 언제 Agentic System을 만들어야 할까?

**"가능한 가장 단순한 방법부터 시작하고, 필요할 때만 복잡도를 높이세요."**

- 많은 경우 LLM API 한 번 호출 + 검색(Retrieval) + 예시(Few-shot)만으로도 충분해요
- Agentic System은 더 나은 성능을 얻는 대신 **지연 시간(Latency)과 비용(Cost)이 증가**해요
- 이 트레이드오프를 신중하게 고려해야 해요

## 2. 기본 구성 요소: 강화된 LLM (Augmented LLM)

모든 Agentic System의 기본 빌딩 블록은 **강화된 LLM**이에요:

```
┌─────────────────────────────┐
│        강화된 LLM            │
│                             │
│  LLM + Retrieval + Tools    │
│        + Memory             │
└─────────────────────────────┘
```

- **Retrieval**: 외부 데이터를 검색해서 LLM에 전달
- **Tools**: LLM이 외부 시스템과 상호작용 (API 호출, DB 조회 등)
- **Memory**: 대화 내용이나 중요 정보를 기억

최신 모델들은 스스로 검색 쿼리를 생성하고, 적절한 도구를 선택하며, 어떤 정보를 기억할지 결정할 수 있어요. [Model Context Protocol (MCP)](https://modelcontextprotocol.io/)을 통해 서드파티 도구 연동도 쉽게 할 수 있어요.

## 3. Workflow 패턴 5가지

### 3.1 Prompt Chaining (순차 연결)

작업을 여러 단계로 나누고, 각 단계 사이에 프로그래밍 방식의 검증(Gate)을 넣는 패턴이에요.

```
입력 → [LLM 호출 1] → 검증 → [LLM 호출 2] → 검증 → 출력
```

**언제 사용하나요?**
- 작업이 명확한 하위 단계로 나뉠 때
- 정확도를 높이기 위해 약간의 지연은 감수할 수 있을 때

**예시:**
- 마케팅 문구 생성 → 다른 언어로 번역
- 문서 개요 작성 → 개요 기반으로 본문 작성

---

### 3.2 Routing (라우팅)

입력을 분류하고, 유형에 맞는 전문 처리기로 보내는 패턴이에요.

```
         ┌→ [전문 처리 A]
입력 → [분류] ─┼→ [전문 처리 B]
         └→ [전문 처리 C]
```

**언제 사용하나요?**
- 입력 유형에 따라 다른 처리가 필요할 때
- 관심사를 분리하고 각각에 최적화된 Prompt를 쓰고 싶을 때

**예시:**
- 고객 문의 유형별 분류 (결제, 환불, 기술 지원 등)
- 간단한 질문은 Claude Haiku, 복잡한 질문은 Claude Sonnet으로 라우팅

---

### 3.3 Parallelization (병렬 처리)

여러 LLM을 동시에 실행하고 결과를 합치는 패턴이에요. 두 가지 변형이 있어요:

**Sectioning (분할):** 작업을 독립적인 부분으로 나눠 병렬 처리
```
       ┌→ [LLM: 파트 A] ─┐
입력 ──┼→ [LLM: 파트 B] ──┼→ 결과 합산
       └→ [LLM: 파트 C] ─┘
```

**Voting (투표):** 같은 작업을 여러 번 실행해서 다양한 관점 확보
```
       ┌→ [LLM: 시도 1] ─┐
입력 ──┼→ [LLM: 시도 2] ──┼→ 다수결/종합
       └→ [LLM: 시도 3] ─┘
```

**예시:**
- Guardrail로 사용자 입력 검증 (보안 체크와 응답 생성을 동시에)
- 코드 리뷰를 여러 관점에서 수행 (보안, 성능, 가독성)
- 콘텐츠 적절성을 다수 평가자로 판단

---

### 3.4 Orchestrator-Workers (오케스트레이터-워커)

중앙 LLM(Orchestrator)이 작업을 동적으로 분해하고, Worker LLM들에게 위임한 뒤 결과를 종합하는 패턴이에요.

```
                 ┌→ [Worker A] ─┐
입력 → [Orchestrator] ─┼→ [Worker B] ──┼→ [결과 종합]
                 └→ [Worker C] ─┘
```

**Parallelization과의 차이점:**
- Parallelization: 하위 작업이 **미리 정해져** 있음
- Orchestrator-Workers: 하위 작업이 **입력에 따라 동적으로** 결정됨

**예시:**
- 코딩 도구가 여러 파일에 걸친 복잡한 변경을 수행
- 검색 작업이 여러 소스에서 정보를 수집하고 분석

---

### 3.5 Evaluator-Optimizer (평가자-최적화자)

하나의 LLM이 응답을 생성하고, 다른 LLM이 평가하고 피드백하는 반복 루프 패턴이에요.

```
입력 → [생성 LLM] → 결과 → [평가 LLM] → 피드백
          ↑                              │
          └──────── 개선 요청 ────────────┘
          (기준 충족 시 종료)
```

**언제 사용하나요?**
- 명확한 평가 기준이 있을 때
- 반복적 개선이 측정 가능한 가치를 제공할 때

**예시:**
- 문학 번역에서 뉘앙스를 반복 개선
- 복잡한 검색에서 여러 라운드에 걸쳐 분석 정제

## 4. 자율 Agent (Autonomous Agent)

위의 패턴들이 정형화된 Workflow라면, Agent는 **LLM이 스스로 판단하며 도구를 사용하는 루프** 형태예요.

```
┌────────────────────────────────┐
│                                │
│  사용자 지시 → [LLM] → 행동   │
│                  ↑       │     │
│                  │       ↓     │
│               피드백 ← 환경    │
│                                │
│  (목표 달성 또는 중단 조건까지 │
│   이 루프가 반복됨)           │
│                                │
└────────────────────────────────┘
```

### Agent가 효과적인 조건

LLM의 다음 능력이 향상될수록 Agent가 더 잘 작동해요:

- 복잡한 입력 이해
- 추론 및 계획 수립
- 도구 사용의 안정성
- 오류 복구 능력

### Agent 구현의 핵심 요소

1. **환경 피드백**: 매 단계마다 도구 실행 결과, 코드 실행 결과 등을 받아야 해요
2. **Human Checkpoint**: 중요한 결정 시점에서 사람의 확인을 받아야 해요
3. **중단 조건**: Agent가 무한 루프에 빠지지 않도록 제어해야 해요

### 언제 Agent를 사용할까?

| 적합한 경우 | 부적합한 경우 |
|-------------|--------------|
| 개방적인 문제 | 정해진 순서가 있는 작업 |
| 단계 수를 예측하기 어려움 | 단순히 한두 단계로 끝남 |
| 고정된 경로로 처리 불가 | 명확한 Workflow로 정의 가능 |

**주의사항:** Agent의 자율성은 신뢰할 수 있는 환경에서 확장성을 제공하지만, **비용이 더 들고 오류가 누적될 수 있어요**. 충분한 Sandbox 테스트와 적절한 Guardrail이 필수예요.

**예시:**
- SWE-bench 과제를 해결하는 코딩 Agent
- Computer Use를 활용한 자동화

## 5. 실전 적용 사례

### 5.1 고객 지원 (Customer Support)

- 챗봇 인터페이스 + 도구 연동의 조합
- Agent가 대화를 관리하면서 고객 데이터, 주문 내역, 지식 베이스에 접근
- 성공 지표가 명확해서 측정하기 좋음 (해결률, 만족도 등)
- 일부 기업은 **성공적인 해결 건에 대해서만 과금**하는 모델을 적용 — Agent에 대한 신뢰 수준이 높다는 뜻

### 5.2 코딩 Agent (Coding Agent)

- 코드는 자동화된 테스트로 검증 가능 → 반복 개선에 유리
- 테스트 결과를 피드백으로 활용하여 Agent가 점진적으로 코드를 개선
- **자동 테스트가 기능 검증**을 하지만, **사람의 리뷰**는 전체 시스템 요구사항과의 정합성 확인에 여전히 필수
- 실제 GitHub Issue를 Pull Request 설명만 보고 해결하는 수준까지 도달

## 6. 도구 설계의 모범 사례

Agent에게 제공하는 도구(Tool)의 설계는 **전체 Prompt만큼이나 중요**해요. Anthropic은 SWE-bench Agent를 개발할 때 전체 Prompt 최적화보다 **도구 최적화에 더 많은 시간**을 투자했다고 해요.

### 도구 형식 (Format) 선택

- 모델이 충분히 추론할 수 있는 토큰을 제공하세요
- 인터넷에서 자연스럽게 나타나는 형식에 가깝게 유지하세요
- 포맷 오버헤드를 피하세요 (정확한 줄 번호 세기, 문자열 이스케이프 등은 LLM에게 어려움)

### 도구 인터페이스 설계 원칙

Agent-Computer Interface(ACI)를 **Human-Computer Interface(HCI)처럼** 설계하세요:

| 원칙 | 설명 |
|------|------|
| **명확한 설명** | 도구의 용도, 매개변수, 사용법을 명확하게 기술 |
| **예시 포함** | 사용 예시, Edge Case, 입력 요구사항, 명확한 경계 |
| **직관적 네이밍** | 매개변수 이름과 설명이 자명하도록 |
| **충분한 테스트** | 다양한 예시로 모델이 도구를 잘 사용하는지 검증 |
| **Poka-yoke** | 실수를 구조적으로 어렵게 만드는 설계 |

### 실전 팁: 절대 경로 사례

> SWE-bench Agent 개발 중, Agent가 루트 디렉토리를 벗어난 후 상대 경로로 파일을 찾지 못하는 실수가 반복됐어요.
> 해결책: **절대 경로를 필수로 요구**하도록 도구를 변경 → 문제가 완전히 해결됐어요.

이처럼 도구 설계 하나만 바꿔도 Agent의 성능이 크게 달라질 수 있어요.

## 7. Agent 성공을 위한 3가지 핵심 원칙

```
┌─────────────────────────────────────────────┐
│                                             │
│  1. 단순함 (Simplicity)                      │
│     → 가장 단순한 해결책부터 시작하세요       │
│                                             │
│  2. 투명성 (Transparency)                    │
│     → Agent의 계획 과정을 명시적으로 보여주세요│
│                                             │
│  3. 도구 품질 (Tool Quality)                  │
│     → 도구 문서화와 테스트에 충분히 투자하세요 │
│                                             │
└─────────────────────────────────────────────┘
```

## 8. 패턴 선택 가이드

어떤 패턴을 써야 할지 결정하는 데 도움이 되는 요약이에요:

| 상황 | 추천 패턴 |
|------|----------|
| 작업이 명확한 단계로 나뉨 | Prompt Chaining |
| 입력 유형별로 다른 처리 필요 | Routing |
| 독립적인 하위 작업이 여러 개 | Parallelization |
| 하위 작업이 입력에 따라 달라짐 | Orchestrator-Workers |
| 반복 개선으로 품질 향상 가능 | Evaluator-Optimizer |
| 개방적이고 단계가 불확실한 문제 | Autonomous Agent |
| 위 모두 과한 경우 | 단일 LLM 호출 + Retrieval |

---

## 요약

> **성공은 가장 정교한 시스템을 만드는 것이 아니라, 필요에 맞는 올바른 시스템을 만드는 것이에요.**

1. 단순한 Prompt부터 시작하세요
2. 포괄적으로 최적화하세요
3. 단순한 방법이 부족할 때만 다단계 Agentic System을 추가하세요
4. 단순함, 투명성, 도구 품질을 유지하면 강력하고 신뢰할 수 있는 Agent를 만들 수 있어요
